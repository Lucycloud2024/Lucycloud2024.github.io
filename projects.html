Spring PetClinic Microservices — Docker & Compose

DMI Cohort 2 — Team Project (App / Docker Lead) + Personal Deployment

🏆 Week 13 Champion Certificate

Situation: An 8-service Spring Boot microservices application needed to be containerised, built, and deployed with full observability — by a team of 11 engineers working across 6 disciplines.

Action: As App/Docker Lead I wrote multi-stage Dockerfiles for all 8 services, built images using the Maven buildDocker profile targeting linux/amd64 for AWS EKS compatibility, fixed a YAML syntax error in docker-compose.yml, implemented proper healthcheck-based startup ordering, and tagged all images with Git SHA for traceability. I also set up the full observability stack — Prometheus scraping metrics every 15 seconds, Grafana dashboards, and Zipkin distributed tracing.

Result: All 8 Docker images built successfully for linux/amd64. Full stack deployed with one command: docker compose up -d. All 11 containers running healthy including Prometheus, Grafana, and Zipkin. Images pushed to AWS ECR for Kubernetes deployment.

Kubernetes Fundamentals Lab

Week 13 — DMI Cohort 2 Assignment

Situation: Needed to demonstrate practical knowledge of core Kubernetes concepts — Pods, ReplicaSets, Deployments, autoscaling, and health probes — in a local cluster environment.

Action: Created a kind cluster (dev-cluster) using kindest/node:v1.31.0 on WSL2 after resolving a kubelet startup failure caused by systemd not being enabled. Deployed Pods using both imperative and declarative methods. Created ReplicaSets with auto-healing demonstration. Built Deployments with RollingUpdate strategy, performed live image upgrades from nginx:1.21.1 to nginx:1.23.1, and executed rollbacks. Configured Horizontal Pod Autoscaler targeting 50% CPU with metrics-server. Added readiness and liveness probes and verified their behaviour by intentionally breaking and fixing them.

Result: Successfully demonstrated all K8s concepts with screenshot evidence. Resolved 2 critical errors (kubelet cgroup issue, kind node image version incompatibility). NodePort and LoadBalancer services configured and tested.

EpicBook Production Deployment — Azure DevOps

Week 11 — Azure CI/CD Pipeline Project

Situation: Deploy a Node.js bookstore web application (EpicBook) to Azure with fully automated infrastructure provisioning and application deployment pipelines.

Action: Built two Azure DevOps pipelines — one for infrastructure (Terraform) and one for application deployment (Ansible). Observability Stack — Prometheus, Grafana, Zipkin

EPIC-5 — DMI Cohort 2 Observability Track

Situation: The Spring PetClinic team needed full observability — metrics collection, dashboard visualisation, distributed tracing, authentication enforcement, and alerting rules.

Action: Configured Prometheus to scrape all 5 petclinic service endpoints every 15 seconds. Verified Grafana pre-provisioned dashboard showing live metrics. Configured Zipkin to receive distributed traces from all services via MANAGEMENT_ZIPKIN_TRACING_ENDPOINT. Enforced Grafana authentication by setting GF_AUTH_ANONYMOUS_ENABLED=false — verified with curl returning HTTP 401 without credentials. Created alert.rules.yml with PetClinicServiceDown, HighErrorRate, and SlowResponseTime alerts. Rebuilt Prometheus image with rules and tested by stopping vets-service — alert fired within 30 seconds.

Result: All 5 OBS tasks completed. Prometheus showing all targets UP. Grafana dashboard with live graphs. Zipkin traces visible across 4 services. Alerts firing correctly on service failure.

Docker Multi-Stage Build — EpicBook Node.js App

Week 12 — Docker Capstone Assignment

Situation: Containerise a Node.js bookstore application with a MySQL backend using Docker best practices including multi-stage builds, network isolation, named volumes, and health checks.

Action: Built a multi-stage Dockerfile that used a Node.js build stage and a lightweight runtime stage to reduce image size. Created a docker-compose.yml with a frontend network, backend network, named volumes for MySQL persistence, health checks for database readiness, and Nginx as a reverse proxy routing traffic from port 80 to the Node.js application on port 3000. Fixed v1 vs v2 docker-compose syntax issues and resolved port 80 conflicts.

Result: Application running with three-tier architecture — Nginx proxy, Node.js app, MySQL database — all in isolated containers. Data persisted across restarts. Database seeded with 54 books.
